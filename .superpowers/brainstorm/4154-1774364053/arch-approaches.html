3 Architecture Approaches

How to structure the system — this affects everything: developer experience, deployment, scalability

A

Monolith — Everything in One FastAPI Service

RAG pipeline, agent execution, connector logic, and WebSocket streaming all live in a single Python service. One Docker container, one port (9001).

┌─────────────────────────────────────┐
│ FastAPI :9001 │
│ ├── /ingest (RAG pipeline) │
│ ├── /query (RAG search) │
│ ├── /agents/* (all 4 agents) │
│ ├── /ws (WebSocket stream) │
│ └── /connectors (demo/real switch) │
└─────────────────────────────────────┘

Pros

  • Simplest deployment
  • No inter-service latency
  • One log stream

Cons

  • Can't scale agents independently
  • One failure takes everything down
  • Large single codebase over time
B

Microservices — Separate Service Per Agent

Each agent runs as its own Docker container. An Orchestrator service coordinates them. RAG backend is separate. 8+ services total.

┌──────────┐ ┌──────────┐ ┌──────────┐
│ RAG :9001│ │Orch:9005 │ │ MCP :9002│
└──────────┘ └──────────┘ └──────────┘
┌──────────┐ ┌──────────┐ ┌──────────┐
│Anomaly │ │Route │ │Notifier │
│:9010 │ │:9011 │ │:9012 │
└──────────┘ └──────────┘ └──────────┘

Pros

  • Scale each agent independently
  • Fault isolated per service
  • True microservices architecture

Cons

  • 8+ services to manage
  • Network overhead between services
  • Over-engineered for current team size
C

Modular Monolith + MCP Connector Layer ⭐ Recommended

Single FastAPI backend with clean internal module boundaries. Agent router, connector abstraction, and streaming all in one service — but structured so any module can be extracted to a microservice later without rewriting. MCP servers stay separate (9002–9004).

Next.js :9000 ──────────────────────────┐
                            ▼
┌────────────── FastAPI :9001 ──────────┐
│ api/rag.py api/agents.py api/ws.py│
│ agents/ connectors/ auth/ │
│ └─ demo.py ←──────→ real.py │
└────────────────────────────────────────┘
      │          │        │
MCP:9002    MCP:9003   MQTT:9005

Pros

  • Simple to deploy now
  • Clean module boundaries = extractable later
  • Per-connector demo/real toggle
  • Single service to debug and log

Cons

  • All agents share one process (acceptable for now)
