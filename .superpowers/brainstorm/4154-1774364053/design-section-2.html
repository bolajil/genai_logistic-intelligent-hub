── AGENT WIRING (the critical fix) ──────────────────
Original problem: agents accepted vector_search_fn + llm_generate_fn but nothing ever passed them in.
Fix: api/agents.py creates real closures from providers and connectors, then calls the agent.
@router.post("/agents/anomaly")
async def run_anomaly_agent(event: AnomalyEvent, user=Depends(auth)):
  # 1. Build real callbacks from providers
  vector_search_fn = lambda q, collection, k: rag.search(q, collection, k)
  llm_generate_fn  = lambda prompt: llm.generate(prompt, trace_id=new_trace())
  # 2. Build data from connector (demo or real)
  shipment = await wms_connector.get_shipment(event.shipment_id)
  # 3. Run agent with real context
  result = agent.respond_to_anomaly(event, shipment, vector_search_fn, llm_generate_fn)
  # 4. Publish alert to SSE stream
  await stream.publish("alerts", result)
  # 5. Log to audit trail
  await audit.log(user, "anomaly_agent", event, result)
  return result
── DATA SIMULATOR (demo mode IoT) ────────────────────
demo/simulator.py — runs as background task when IoT connector = demo
Generates realistic temperature drift, GPS movement, door events for all mock shipments
Publishes to Redis pub/sub → picked up by SSE stream → frontend gauges update live
Injects a temperature breach every ~5 minutes to trigger AnomalyResponder automatically
Configurable: breach_frequency, drift_rate, shipment_count

Key Point — What This Fixes

The original codebase had all 4 agents fully written but completely disconnected — their callbacks were always None. The api/agents.py router is the missing wire. It creates real closures from the actual RAG + LLM providers and injects them when calling each agent. No changes needed to the existing agent classes themselves.
