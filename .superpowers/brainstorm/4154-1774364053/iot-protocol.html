Real-Time IoT Data Protocol

When production connections are approved, how do sensors push temperature/GPS/door data to the platform?

A

REST Polling

Backend polls the IoT API every N seconds to fetch latest sensor readings. Simple, firewall-friendly, works with any existing WMS/TMS REST API.

Backend → GET /sensors/readings (every 30s)
Backend → GET /shipments/TX-CHI-2025-001/temp
Backend processes → alerts if breach

Pros

  • Works with any REST API
  • Simple, debuggable
  • No special infra needed

Cons

  • 30s+ latency on anomalies
  • Not truly real-time
B

WebSocket / SSE (Real-Time Push)

IoT sensors push data to the backend via WebSocket or Server-Sent Events. Frontend dashboard updates live — temperature gauges move in real time. Demo mode simulates this with a data generator.

IoT Sensor → WS push → Backend
Backend → SSE → Frontend (live dashboard)
Anomaly detected → immediate alert (sub-second)

Pros

  • Truly real-time (<1s latency)
  • Live dashboard updates
  • Demo generates live fake data

Cons

  • More complex connection management
  • Requires persistent connections
C

MQTT (Industry IoT Standard)

The protocol used by most enterprise cold chain sensors (Carrier, Sensitech, Tive). Pub/sub model — sensors publish to topics, backend subscribes. Standard in Lineage-scale logistics.

Sensor → MQTT broker → topic: lineage/truck/TX-CHI/temp
Backend subscribes → AnomalyResponder triggered
Demo: mock MQTT broker with simulated sensor stream

Pros

  • Industry standard for cold chain
  • Low bandwidth, designed for IoT
  • What real Lineage systems would use

Cons

  • Requires MQTT broker (Mosquitto/HiveMQ)
  • New protocol vs REST

Recommendation

B + C hybrid: WebSocket/SSE for the frontend dashboard (live gauges, real-time alerts), MQTT connector for production IoT ingestion. Demo mode runs a built-in data simulator that feeds the same WebSocket pipeline. This way the demo looks exactly like production.
