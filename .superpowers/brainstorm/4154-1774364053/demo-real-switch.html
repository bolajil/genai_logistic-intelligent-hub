How should Demo ↔ Production mode switching work?

This is architecturally foundational — affects how connectors, APIs, and the UI are structured

A

Environment Variable Only

Set GLIH_MODE=demo or GLIH_MODE=production in the .env file. No UI toggle — ops team controls it at deployment time. Simple, safe, audit-friendly.

Pros

  • No accidental mode switch in prod
  • Clean infrastructure boundary
  • Simplest to implement

Cons

  • Requires server restart to switch
  • Can't demo both modes live
B

Global UI Toggle (Admin Page)

Admin dashboard has a Mode Switch — one toggle flips the entire platform from Demo to Production. Persisted in the database, no restart needed. Requires admin role to change.

Pros

  • Can switch live during demos
  • Visible to all team members
  • No server restart needed

Cons

  • Risk of accidental flip
  • More complex UI + state management
C

Per-Connection Toggle (Granular)

Each integration — WMS, IoT, Docs — has its own Demo/Real toggle in the Settings page. Mix and match: IoT on real sensors, WMS still on demo. Most flexible for phased rollout.

Pros

  • Phased rollout (approve one at a time)
  • Perfect for client approval workflow
  • Can test real IoT while WMS stays safe

Cons

  • Most complex to build
  • Risk of mixed state confusion

Recommendation

Option C is ideal for a production platform targeting enterprise clients — it mirrors exactly how a real phased rollout works: "approve WMS real connection this sprint, IoT next sprint." But it requires more upfront architecture.
